# showwhat

> Schema and rule-based evaluation engine for feature flags and configuration management.

showwhat is a vendor-neutral, infrastructure-free feature flag library. It resolves feature definitions against runtime contexts using YAML or JSON-based rules. First matching variation wins.

- Documentation: https://showwhat.yeojz.dev
- Repository: https://github.com/nicerapp/showwhat
- License: MIT

## @showwhat/core

Low-level engine. Key exports:

- `resolve(definitions, context, evaluators)` — evaluate all definitions, return resolutions
- `resolveVariation(definition, context, evaluators)` — evaluate a single definition
- `builtinEvaluators` — condition types: env, string, number, bool, datetime, startAt, endAt, and, or
- `registerEvaluators(custom)` — merge custom evaluators with builtins
- `MemoryData` — in-memory data source implementing DefinitionReader/DefinitionWriter
- `parseYaml(str)`, `parseObject(obj)` — parse definitions from YAML strings or plain objects
- `parsePresetsYaml(str)`, `parsePresetsObject(obj)` — parse preset condition templates
- `createPresetConditions(presets, params)` — instantiate reusable condition templates
- Zod schemas: Definition, Variation, Condition, Context, Resolution
- Typed errors: ShowwhatError, ValidationError, DefinitionNotFoundError, VariationNotFoundError, InvalidContextError, DataError, ParseError, SchemaValidationError

Architecture: `schemas/` contains Zod types (no evaluator imports). `conditions/` contains evaluator implementations (imports from schemas). No default evaluators injected; consumers must pass evaluators explicitly.

## showwhat (main package)

High-level convenience API. Re-exports everything from @showwhat/core plus:

- `showwhat({ keys?, context, options: { data } })` — resolve one or more keys, returns `Resolutions` record
  - `keys` provided: fetches each via `data.get(key)` in parallel
  - `keys` omitted: fetches all via `data.getAll()`
  - Per-key errors wrapped in `ResolutionError`, never thrown
  - Only systemic failures (context validation) are thrown
- `ResolutionError` — `{ success: false; key: string; error: ShowwhatError }` for failed keys
- `Resolutions` — `Record<string, Resolution | ResolutionError>`
- `Resolution` has `success: true`, `ResolutionError` has `success: false` — check `entry.success` to narrow
- Validates context against ContextSchema (Zod) automatically

Use `showwhat` for resolving flags with built-in evaluators and error wrapping. Use `@showwhat/core` directly for low-level batch resolution or custom evaluator injection.

## @showwhat/configurator

React component library for visually building definitions. Key exports:

- `Configurator` — main component, accepts a ConfiguratorStore
- `DefinitionEditor`, `DefinitionList` — definition CRUD
- `VariationCard`, `VariationList` — variation editing with drag-drop (@dnd-kit)
- `ConditionBuilder` — visual AND/OR condition nesting
- `ValueInput`, `DateTimeInput` — value editors
- `ConfiguratorStore` interface — apps provide their own persistence backend
- `useConfiguratorStore`, `useActionState`, `useConfiguratorSelector` — hooks
- CSS exported at `@showwhat/configurator/styles.css` (must be imported by consumers)

Design: controlled components (props in, callbacks out), no internal mutations. Store interface abstracts persistence so apps can back it with files, localStorage, or APIs.

## @showwhat/openfeature

OpenFeature Server SDK provider. Key exports:

- `ShowwhatProvider` — implements OpenFeature provider interface
- `toShowwhatContext()` — maps OpenFeature evaluation context to showwhat context
- Supports boolean, string, number, object evaluation types
- Custom evaluators via options, lifecycle management, error code mapping

## Definition Structure

A definition is a feature flag configuration:

```yaml
my-feature:
  description: Optional description
  active: true                    # set false to disable
  variations:
    - value: true
      conditions:                 # first match wins
        - type: env
          op: eq
          value: production
    - value: false                # catch-all (no conditions)
```

Supported condition types: env, string, number, bool, datetime, startAt, endAt, and (nested), or (nested). Custom evaluators can be registered.

## Coding Conventions

- `interface` for contracts/extension points, `type` for data shapes and props
- No enums (use string unions), no `any` (use `unknown`)
- ESM only, no CommonJS
- Comments explain why, not what
- Test files: `*.test.ts` / `*.test.tsx`
- Entry points: `src/index.ts` with controlled exports
- Tailwind classes via `clsx()` and `tailwind-merge`
- Errors use typed hierarchy with context (key, message, type)
